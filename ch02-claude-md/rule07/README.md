# Rule 07: 削除自問テストでプルーニングする

書籍『Claude Code 設計の30の鉄則』 Rule 07 のサンプルです。

## ファイル

- `antipattern/CLAUDE.md`: 自明な原則、推測可能な情報、検証不能な
  スローガンが混在した CLAUDE.md。約 100 行のうち 70 行が削除対象
- `fixed/CLAUDE.md`: 削除自問テストを 1 行ずつ適用した結果。
  約 25 行に圧縮 (詳細規約は paths スコープルールに切り出し)
- `fixed/PRUNING_LOG.md`: どの行を、なぜ削除したかの判断ログ。
  Anthropic Engineering Blog の「Ruthlessly prune」を 4 カテゴリで
  実演する

## 削除自問テストの 4 段階

1. **1 行ずつ読む** (パラグラフではなく行単位)
2. **質問する**: 「この行を削除したら Claude は間違うか?」
3. **NO なら削除**: 自明 / 既知 / 推測可能 / 過去のもの
4. **YES なら昇格を検討**: hook 化、または skills / rules への移動

## 削除候補の典型 4 カテゴリ

- カテゴリ A: Claude が当然知っている言語標準・原則
- カテゴリ B: 自明な good practice
- カテゴリ C: コードから推測可能な情報
- カテゴリ D: 抽象的すぎて検証不能なスローガン

## 残すべき 3 カテゴリ

- カテゴリ E: プロジェクト固有のコマンド (推測不可能)
- カテゴリ F: プロジェクト固有の罠 (非自明な振る舞い)
- カテゴリ G: hook 化候補 (将来 PreToolUse hook に昇格)

## 検証バージョン

Claude Code CLI v2.x
