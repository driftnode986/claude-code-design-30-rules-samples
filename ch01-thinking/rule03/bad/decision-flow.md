# Release Notes Generator (BAD: 立ち位置不明の混在型)

「直近 1 週間分のマージ済み PR からリリースノートを書く」というタスクを処理するエージェント定義。
立ち位置を Workflow にも Agent にも決めずに、Subagent を「便利な関数」として呼んでいる。

## システム構成

メインの Claude Code セッションで以下を実行する:

1. ユーザーが「先週分のリリースノート作って」と言う
2. メイン Claude Code が PR 一覧を取得する (`gh pr list --state merged --search "merged:>=2026-05-10"`)
3. PR ごとに **Subagent を起動**して以下を依頼:
   - 「この PR の diff と description を読んで、3 行のリリースノート項目を書いて」
   - Subagent は内部で gh CLI を呼び diff を取得、自分の判断で要約を返す
4. メインに戻って 30 件分のリリースノート項目を結合する
5. **別の Subagent を起動**して以下を依頼:
   - 「30 件の項目をカテゴリ分けして並び替えて」
   - Subagent は内部でカテゴリを定義し、並び替えロジックを自分で考えて返す
6. **さらに別の Subagent を起動**して以下を依頼:
   - 「リリースノート全体を読みやすく整形して、絵文字も付けて」
7. メインが最終結果を出力

## なぜこれが中途半端か

このフローは Workflow としても Agent としても破綻している。

### Workflow としての破綻

Workflow ならば「定義済みコードパスで LLM とツールを orchestrate」するはずだが、ここでは:

- カテゴリ定義 (5 番目のステップ) を Subagent に委ねている → ステップごとのカテゴリが揺れる
- 「読みやすく整形」「絵文字を付ける」の基準 (6 番目のステップ) も Subagent に委ねている → 週次でフォーマットが揺れる
- 結果として「予測可能性と一貫性」という Workflow の長所が失われている

### Agent としての破綻

Agent ならば「LLM が動的に方向付け、タスク達成方法を制御する」はずだが、ここでは:

- ステップ 1〜7 が事前に固定されていて、Subagent には方向付けの権限がない
- Subagent が「実は PR を 30 件全部処理する必要はない、最も影響度の高い 5 件に絞るべき」という判断をしても、その判断を反映する余地がない
- 結果として「柔軟性とモデル駆動の意思決定」という Agent の長所が失われている

### 観測される症状

- 同じ「先週分」を 2 回流すと、カテゴリ分けと絵文字の付け方が毎回違う (Subagent の自由度が高すぎる)
- 30 PR を順に Subagent に投げるため、Multi-agent の 15 倍トークン消費 (本書 Rule 25) に近い負荷がかかる
- にもかかわらず最終結果の品質は Workflow で組んだ場合とほぼ変わらない (Subagent の動的判断が活きていない)

## どう直すか

立ち位置を **Workflow** に振るなら → `good/workflow-example.md` のようにステップ間の判断基準を確定し、Subagent ではなく単一 LLM 呼び出し (Prompt chaining) で組む。

立ち位置を **Agent** に振るなら → ステップ 1〜7 を固定せず、Claude Code 自身に「どの PR を取り上げ、どう分類し、どう書くか」を委ね、人間は最終レビューだけする (この用途では過剰だが、「バグ調査・修正」のような open-ended タスクなら自然な選択肢)。

このタスクの場合は **Workflow が正解**。リリースノートは予測可能性と一貫性が重要で、柔軟性は本質的な要求ではない。
